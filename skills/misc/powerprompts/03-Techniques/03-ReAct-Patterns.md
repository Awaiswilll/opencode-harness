## What Is ReAct?

**ReAct** (Reasoning + Acting) interleaves chain-of-thought reasoning with tool use in a single loop. Published by Yao et al. (2022), it's the foundational architecture behind virtually every modern AI agent — coding agents, research agents, browser agents, and more.

### The Core Insight

- **Pure CoT** is closed-loop: the model reasons against its own training memory, which hallucinates when knowledge is missing or outdated.
- **Pure tool use** is reactive: the model calls tools but doesn't reason strategically about which tools to call and why.
- **ReAct** combines both: reason → act → observe → reason again. Every reasoning step is grounded in real observations.

### Historical Impact

| Before ReAct | After ReAct |
|-------------|-------------|
| LLMs answer from parametric knowledge only | LLMs can search, compute, and verify |
| Hallucinations are unchecked | Actions provide ground truth |
| Single-turn Q&A | Multi-turn autonomous agents |
| No way to correct mistakes | Observations enable self-correction |

> *"ReAct collapses the hallucination surface by grounding every reasoning step in real observations."*

## The Thought → Action → Observation Loop

### The Basic Loop

```
Question: [user query]
Thought: [internal reasoning about what to do and why]
Action: [tool_name(arguments)]
Observation: [result from the tool]

Thought: [reasoning about the observation]
Action: [next tool call or final answer]
...

Final Answer: [synthesized response grounded in observations]
```

### Each Component Explained

| Component | Purpose | Rules |
|-----------|---------|-------|
| **Thought** | Reasoning, planning, analysis | Must precede every Action. Can be: "I need to find X", "The result shows Y, so now I need Z" |
| **Action** | A specific tool invocation | Format: `tool_name(arg1="value", arg2="value")`. One action per step. |
| **Observation** | The result of the action | Provided by the runtime, NOT generated by the model. Contains tool output or error. |
| **Final Answer** | The synthesized response | Only emit when sufficient information is gathered. Provides answer with citations. |

### The Feedback Cycle

```
    ┌─────────────────────────────────────────┐
    │                                         │
    ▼                                         │
Thought ──→ Action ──→ Observation ──→ Thought
                                           │
                                           │ (if enough info)
                                           ▼
                                      Final Answer
```

Each loop grounds the model's reasoning in external reality. The model can:
- Verify its assumptions
- Correct course when observations contradict expectations
- Accumulate evidence across multiple tool calls

## Tool-Use Integration

### Tool Definition Format

Tools must be clearly defined in the system prompt:

```
You have access to the following tools:

search(query: str) → list of {title, url, snippet}
  Description: Search the web for current information
  Use when: You need recent data, facts, or sources

calculator(expression: str) → number
  Description: Evaluate a mathematical expression
  Use when: You need precise calculations

read_page(url: str) → str
  Description: Fetch and extract text from a URL
  Use when: You need full context from a specific source

execute_python(code: str) → dict
  Description: Run Python code and return stdout/stderr
  Use when: You need computation, data analysis, or visualization
```

### Runtime Protocol

```
1. Model emits: Action: tool_name(arguments)
2. Runtime parses the action, executes the tool
3. Runtime appends: Observation: [result]
4. Runtime sends the full conversation back to the model
5. Model continues with Thought → Action or Final Answer
```

### Action Space Design Principles

| Principle | Good | Bad |
|-----------|------|-----|
| Focused tools | `search`, `calculate`, `read` | 20+ overlapping tools |
| Clear descriptions | "Use for current events" | "A search tool" |
| Consistent naming | snake_case for all tools | mixCase, camelCase, kebab-case |
| Fail-safe tools | Read-only tools preferred | Destructive tools without safeguards |

### Tool Selection Heuristics

Include rules in the system prompt to guide tool selection:

```
- Use search() when you need information not in your training data
- Use calculator() for all arithmetic (don't do math in your head)
- Use read_page() to get full context from search results
- If a tool fails, try an alternative approach ONCE before giving up
- Never fabricate tool results — if you didn't call the tool, don't pretend
```

## Complete Example: Research Agent

### Scenario

Research the impact of quantum computing on cryptography and produce a structured report.

### System Prompt

```
You are a research agent with these tools:
- search(query) → web search results (title, url, snippet)
- read_page(url) → full text content
- summarize(text) → condensed summary

Process:
1. For each major subtopic: search → read → summarize
2. Cover 3+ distinct subtopics
3. Synthesize findings into a structured report
4. Never fabricate sources
5. If search returns nothing, note it explicitly

Return format:
## Report: [title]

### Overview
[2-3 sentence overview]

### Key Findings
- [finding 1] (source: [url])
- [finding 2] (source: [url])

### Sources
- [source 1]
- [source 2]
```

### Execution Trace

```
User: Research the impact of quantum computing on cryptography.

Thought: This is a broad topic. I'll break it into subtopics:
1. Current state of quantum computing for cryptanalysis
2. Post-quantum cryptography standards
3. Timeline estimates for quantum threats
Let me start with the first subtopic.

Action: search("quantum computing impact on cryptography 2026")

Observation: [Search results about Shor's algorithm, RSA vulnerability,
post-quantum cryptography NIST standards, timeline estimates...]

Thought: The search returned relevant results. Let me read the most
authoritative source about current quantum computing capabilities
for breaking encryption.

Action: read_page("https://example.com/quantum-cryptography-2026")

Observation: [Full article text covering:
- Current quantum computers: ~1000 logical qubits
- RSA-2048 estimated to need ~20M qubits (not yet feasible)
- NIST post-quantum standardization: 4 algorithms selected
- Timeline: 15-20 years before quantum threat is real
...]

Thought: Good, I have solid information about the current state.
Now let me research the second subtopic: post-quantum standards.

Action: search("NIST post-quantum cryptography standards 2026")

Observation: [Results about CRYSTALS-Kyber, CRYSTALS-Dilithium,
SPHINCS+, FALCON selection...]

[Continues for 3-5 more iterations...]

Thought: I now have comprehensive information across three subtopics.
Let me synthesize the report.

Final Answer:
## Report: The Impact of Quantum Computing on Cryptography

### Overview
Quantum computing poses a fundamental threat to current public-key
cryptography. While large-scale quantum computers capable of breaking
RSA-2048 are not yet feasible, the NIST has standardized four
post-quantum cryptographic algorithms, and organizations are urged
to begin migration planning now.

### Key Findings
- **Current capability gap**: Today's quantum computers (~1000 logical
  qubits) are far from the ~20M qubits needed to break RSA-2048
  (source: https://example.com/quantum-cryptography-2026)
- **NIST standards finalized**: CRYSTALS-Kyber (encryption) and
  CRYSTALS-Dilithium (signatures) selected as primary standards
  (source: https://example.com/nist-pqc-2026)
- **Migration timeline**: Experts recommend beginning crypto-agile
  migration by 2028, with full transition by 2035
  (source: https://example.com/pqc-migration-timeline)

### Sources
- https://example.com/quantum-cryptography-2026
- https://example.com/nist-pqc-2026
- https://example.com/pqc-migration-timeline
```

## Complete Example: Coding Agent

### Scenario

Debug a failing test and fix the implementation.

### System Prompt

```
You are a coding agent with these tools:
- read_file(path) → file contents
- grep_file(pattern, path) → search results
- write_file(path, content) → write changes
- run_test(test_name) → test output
- git_diff() → current changes

Process:
1. Read the failing test to understand expected behavior
2. Search for the relevant source code
3. Diagnose the issue using test output
4. Implement the fix
5. Verify with tests

Rules:
- Read before writing — understand the full context first
- Run tests after every change
- If tests still fail after 3 attempts, explain what you've learned
```

### Execution Trace

```
User: The test test_calculate_discount() is failing. Fix it.

Thought: Let me first read the failing test to understand what it expects.

Action: read_file("tests/test_pricing.py")

Observation: [File shows test_calculate_discount:
def test_calculate_discount():
    assert calculate_discount(100, "STANDARD") == 10
    assert calculate_discount(100, "VIP") == 20
    assert calculate_discount(100, "EMPLOYEE") == 35
]

Thought: The test expects different discount percentages based on
customer type. Let me find the implementation.

Action: grep_file("def calculate_discount", "src/")

Observation: [src/pricing.py:23: def calculate_discount(total, customer_type):]

Action: read_file("src/pricing.py")

Thought: Let me see the current implementation...

Observation: [src/pricing.py:
def calculate_discount(total, customer_type):
    if customer_type == "VIP":
        return total * 0.2
    return total * 0.1
]

Thought: Found the bug. The implementation returns float values
(20.0, 10.0) but the test expects integers (20, 10). Also, the
EMPLOYEE type is missing entirely. Let me fix both issues.

Action: write_file("src/pricing.py", "def calculate_discount(total, customer_type):
    if customer_type == "EMPLOYEE":
        return int(total * 0.35)
    if customer_type == "VIP":
        return int(total * 0.2)
    return int(total * 0.1)
")

Observation: File written successfully.

Thought: Now let me verify the fix passes the test.

Action: run_test("test_calculate_discount")

Observation: [Test passed! ✓]

Final Answer: Fixed the `calculate_discount` function in
`src/pricing.py:23` — added int() wrappers for integer return
values and added the missing EMPLOYEE discount tier (35%).
```

## Complete Example: Detective Analysis

### Scenario

Compare population densities of Tokyo and Paris.

```
Question: What is the population density of Tokyo vs Paris?

Thought: I need population and area data for both cities to
calculate density. Let me start with Tokyo.

Action: search("Tokyo population 2025 city proper")

Observation: Tokyo has 13.96 million people (city proper).

Thought: Now I need Tokyo's area.

Action: search("Tokyo area km2")

Observation: Tokyo covers 2,194 km2.

Thought: Tokyo density = 13.96M / 2,194 = ~6,363/km2.
Now let me get the same for Paris.

Action: search("Paris population 2025 city proper")

Observation: Paris has 2.16 million people (city proper).

Thought: And Paris area...

Action: search("Paris area km2")

Observation: Paris covers 105 km2.

Thought: Paris density = 2.16M / 105 = ~20,571/km2.
Let me verify these calculations.

Action: calculator("13960000 / 2194")

Observation: 6362.81

Action: calculator("2160000 / 105")

Observation: 20571.43

Thought: The numbers check out. Let me verify I have the right
geographical definitions — city proper vs metro area.

Action: search("Tokyo metro area population 2025")

Observation: Greater Tokyo Area: ~37.8 million (37,274 km2)
→ 37.8M / 37,274 = ~1,014/km2

Thought: Important distinction. City proper vs metro area gives
very different densities. I'll provide both for context.

Final Answer:
- **Tokyo city proper**: ~6,363/km² (13.96M / 2,194 km²)
- **Paris city proper**: ~20,571/km² (2.16M / 105 km²)
- **Greater Tokyo metro**: ~1,014/km² (37.8M / 37,274 km²)
- **Paris metro area**: ~3,900/km² (13M / 2,845 km²)

Paris city proper is ~3.2x denser than Tokyo city proper.
However, the Tokyo metro area covers a much larger region
with lower overall density.
```

## Graceful Failure Handling

### Failure Mode 1: Tool Returns Nothing

```
Action: search("latest quantum computing breakthrough 2026")

Observation: No results found.

❌ Bad: "Search results show that quantum computing is advancing..."
✅ Good (Strategy 1 — Broaden): "No direct results. Let me try broader terms."
✅ Good (Strategy 2 — Verify): "Let me try alternative search terms."

Action: search("quantum computing news 2026")
```

### Failure Mode 2: Tool Errors

```
Action: read_page("https://example.com/broken-link")

Observation: Error: HTTP 404 — page not found.

❌ Bad: [Pretends to have read the page]
✅ Good: "The page returned a 404. Let me search for alternative sources."
```

### Failure Mode 3: Contradictory Information

```
Observation (from source A): "Quantum computers will break RSA by 2030."
Observation (from source B): "Quantum computers won't break RSA until 2045+."

✅ Good: "I found conflicting claims. Source A estimates 2030,
source B estimates 2045+. I'll report both and note the disagreement
rather than picking one."
```

### Failure Mode 4: Complete Dead End

```
After 3 attempts with different approaches, no useful information found.

✅ Good: "After searching multiple sources, I was unable to find
reliable information about [topic]. Available approaches I tried:
1. [approach 1 — no results]
2. [approach 2 — irrelevant results]
3. [approach 3 — inaccessible sources]

I recommend: [alternative suggestion for finding the information]"
```

### System-Level Graceful Failure Template

Include in the ReAct system prompt:

```
FAILURE PROTOCOL:
- If a tool returns no results → try 2 alternative approaches
- If a tool errors → search for the information through a different path
- If you encounter contradictory information → report all perspectives
- If you cannot complete the task after 3 attempts → provide:
  1. What you attempted
  2. What you learned (even partial)
  3. Why you couldn't complete it
  4. Alternative approaches you recommend
```

## System Prompt Design for ReAct

### Production-Grade ReAct System Prompt

```
You are a ReAct agent. For each step, you MUST alternate between:

- Thought: Your internal reasoning about what to do and why
- Action: A tool call in the format tool_name(arguments)
- Observation: The result of your action (provided by the system)

Continue this cycle until you have enough information.
Then respond with: Final Answer: [your answer]

Available tools:
{tools}

RULES:
1. Every Action MUST be preceded by a Thought
2. Every Action MUST be followed by an Observation
3. Do NOT fabricate observations — only use what the system provides
4. If stuck after 3 attempts, admit inability
5. Use calculator() for ALL arithmetic — do not compute in your head
6. Cite sources in Final Answer using [source_number] notation

THOUGHT GUIDELINES:
- Start with: "I need to find [what information]"
- After observation: "This tells me [insight]. Now I need [next step]"
- For verification: "Let me verify [claim] by [method]"
- When stuck: "Approach [X] failed. Let me try [alternative Y]."
```

### Minimal ReAct Prompt

For simpler agents:

```
Solve the task by alternating reasoning and tool use.

Thought: [reasoning]
Action: [tool]
Observation: [result]
...repeat...
Final Answer: [answer]

Tools: {tools}
```

## ReAct vs Pure CoT vs Tool-Use Only

| Capability | Pure CoT | Tool-Only | ReAct |
|-----------|----------|-----------|-------|
| Reasoning | ✅ Strong | ❌ None | ✅ Strong |
| Grounding | ❌ Hallucinates | ✅ Always factual | ✅ Grounded |
| Tool selection | ❌ N/A | ✅ Use tools | ✅ Strategic use |
| Error recovery | ❌ Linear | ⚠️ Limited | ✅ Loop with feedback |
| Multi-step plans | ✅ Can plan | ⚠️ Reactive | ✅ Plan + execute |
| Cost | Low | Medium | High (but most capable) |
| Implementation | Easy | Medium | Complex |

### When to Use Each

| Use ReAct When | Use CoT When | Use Tool-Only When |
|----------------|-------------|-------------------|
| Task requires external data | Pure reasoning task | Simple lookup tasks |
| Facts need verification | Math/logic problems | Single API call |
| Multi-step research | Creative writing | Data extraction |
| Grounding is critical | Analysis | Format conversion |

## Production Implementation

### ReAct Loop in Python

```python
import re

def react_loop(question, tools, max_iterations=10):
    messages = [{"role": "user", "content": question}]
    system_prompt = build_react_system_prompt(tools)

    for _ in range(max_iterations):
        response = llm.generate(messages, system=system_prompt)

        # Check for final answer
        if "Final Answer:" in response:
            return extract_final_answer(response)

        # Parse action
        action_match = re.search(r"Action: (\w+)\((.*)\)", response)
        if not action_match:
            # Model is reasoning but not acting — prompt or retry
            messages.append({"role": "assistant", "content": response})
            messages.append({"role": "user",
                "content": "Please specify an Action or provide Final Answer."})
            continue

        tool_name = action_match.group(1)
        tool_args = parse_args(action_match.group(2))
        tool_func = tools.get(tool_name)

        if not tool_func:
            observation = f"Error: Tool '{tool_name}' not found."
        else:
            try:
                result = tool_func(**tool_args)
                observation = str(result)
            except Exception as e:
                observation = f"Error: {str(e)}"

        messages.append({"role": "assistant", "content": response})
        messages.append({"role": "user",
            "content": f"Observation: {observation}"})

    return "Max iterations reached without final answer."
```

### Key Implementation Details

| Detail | Why |
|--------|-----|
| Parse `Action:` markers | Extract structured calls from free-form text |
| Execute in try/except | Tool failures shouldn't crash the agent |
| Append observations as user messages | Preserves conversation structure |
| Limit iterations | Prevent infinite loops (10 is usually enough) |
| Handle no-action cases | Sometimes the model forgets to act |

## Advanced Patterns

### 1. Parallel ReAct

Execute multiple independent tool calls simultaneously to reduce latency:

```
Thought: I need population and area for both cities. These are independent.

Action 1: search("Tokyo population 2025")
Action 2: search("Tokyo area km2")
Action 3: search("Paris population 2025")
Action 4: search("Paris area km2")
```

### 2. Reflective ReAct

Add a reflection step after every N actions to assess overall progress:

```
Thought (Reflection): I've taken 3 actions so far. Let me assess:
- What I know: Tokyo population (13.96M), Tokyo area (2,194 km²)
- What I still need: Paris data
- Am I on track? Yes, I've completed ~50% of the research
- Should I change strategy? No, current approach is working
```

### 3. Tool-Chaining ReAct

Chain tool calls where the output of one feeds the next:

```
Thought: I need to find recent stock data.

Action: search("AAPL stock price 2026")

Observation: [results showing current price ~$245]

Thought: Now let me read a detailed analysis.

Action: read_page("https://finance.example.com/AAPL")

Observation: [detailed analysis...]

Thought: Let me extract the key metrics.

Action: summarize("${observation}")

Observation: [structured summary...]
```

### 4. Human-in-the-Loop ReAct

```
Thought: I need to delete the production database to fix this issue.
Wait — this is dangerous. I should ask for confirmation.

Action: ask_user("I found that deleting table 'users_temp' would
resolve the issue. Can I proceed? (yes/no)")

Observation: User says: "No, find another approach."

Thought: The user declined. Let me find a non-destructive
alternative. I could add a migration instead.
```

## Common Failure Modes

| Failure Mode | Symptom | Fix |
|-------------|---------|-----|
| Hallucinated observations | Model claims tool result without calling tool | Strict system prompt: "Never fabricate observations" |
| Action loops | Same action repeated without progress | Add loop detection, max iterations |
| No action after thought | Model reasons but never acts | Prompt: "What tool will you use?" |
| Wrong tool selection | Uses search when calculator needed | Better tool descriptions |
| Verbose thoughts | 80% of tokens in reasoning | Set thought length limit |
| Premature final answer | Answers before sufficient evidence | Enforce minimum action count |
